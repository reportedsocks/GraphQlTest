GraphQL 

| Pros                                                                                                                                | Cons                                                                                                |
|-------------------------------------------------------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------------------------|
| Бекенд робить схему, клієнт отримує гнучкі опції для запитів з одного ендпоінту                                                     | Більш складний сетап в порівнянні з REST. Зі сторони бекенду складно сказати(?)                     |
| No over fetching. Якщо якесь поле зайве, то можна виключити його з запиту без участі бекенду                                        | В цілому можна отримати теж саме через фільтр атрибутів в REST                                      |
| No under fetching. Якщо треба витягнути об'єкт і його зв'язки, то можна зробити це одним запитом                                    | Якщо сам бекенд виконує окремий запит на зв'язки об'єкта,то час очікування клієнта може збільшитись |
| Декілька клієнтів (платформ) можуть взаємодіяти з бекендом так як їм потрібно -> менше дзвінків і тікетів "оновити назву параметру" | Якщо клієнт лише 1, або іос і андроїд мають ідентичний функціонал, то не так вже і корисно          |
| Генерація DTO                                                                                                                       | Мінусів не виявлено :)                                                                              |

Як висновок, то найкращі умови для GraphQL:

- Мобайл + веб платформи на клієнтах
- Сервер має власну БД з різними зв'язками. Тобто якщо це хаб з різних АПІ, то бенефіти туманні
- Розмір проекта і команди >= середній
- Є час на підтримку схеми, онбординг нових людей в команду
- Якийсь кейс вайтлейбл апі(?)
